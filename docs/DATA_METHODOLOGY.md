# 데이터 방법론 (Data Methodology)

## 1. 데이터 수집

139개 이상의 종 데이터를 아래 방식으로 수집했습니다.

- **1차 소스**: 뉴질랜드 아쿠아리움 판매점(legacyaquatics.co.nz) 상품 페이지 — 실제 판매 중인 종의 스펙(온도/pH/크기) 확인
- **2차 소스**: FishLore.com, AquaticCommunity.com — 종 프로필 + 실사용자 코멘트(사육 경험담)
- 저작권을 고려해 **수치(사실 정보)만 추출**하고, 설명 문장은 자체적으로 재작성했습니다. 특정 사이트를 자동으로 대량 수집(크롤링)하지 않고, 종 단위로 필요한 페이지만 개별 확인하는 방식을 택했습니다.

## 2. 데이터 모델링 — 왜 그룹/종을 분리했나

초기에는 종 하나당 스펙을 전부 갖는 flat 구조였습니다. 그런데 실제 사육 시장에는 **같은 학명, 다른 색상**인 품종(구피, 시클리드 색변이 등)이 매우 많아, flat 구조로는 같은 스펙을 수십 번 중복 입력해야 했습니다.

→ `careGroups`(스펙) / `species`(개체 식별 정보) 두 테이블로 분리하고, `species.groupId`로 참조하는 **1:N 관계형 구조**로 리팩터링했습니다.

이 과정에서 실수도 있었습니다 — 처음에는 "생김새가 비슷하다"는 이유로 서로 다른 속(genus)의 종을 같은 그룹의 색상 변이(alias)로 잘못 묶은 사례가 있었고, 이후 아래 검증 규칙을 도입했습니다.

> **규칙**: `aliases`에는 반드시 같은 학명(genus + species)의 색상/품종 변이만 넣는다. 속이 다르거나 학명이 불확실하면 무조건 별도 `species` 항목으로 분리하고, 확신이 없으면 "확인 필요" 플래그를 붙인다.

## 3. 다중 소스 교차검증

동일 종에 대해 서로 다른 소스의 수치가 다를 경우:

1. 소스가 오래되었거나(예: 2004~2008년 콘텐츠) 단일 사례 기반이면 낮은 신뢰도로 취급
2. 여러 독립 소스가 수렴하는 값을 채택
3. 생물학적 기초 지식과 모순되는 수치는 출처와 무관하게 배제
   - 예: 특정 소스가 말라위 시클리드(Demasoni)의 적정 pH를 6.5~7.5로 제시했으나, 말라위호 자체가 강알칼리성(7.7~8.6)이라는 기초 생태 지식과 모순되어 채택하지 않음

이 과정에서 `scripts/verify_against_source.py`로 소스 간 종 목록을 매칭했고, **"같은 속이면 매칭"이라는 느슨한 규칙이 오탐을 만든다**는 걸 확인해 "같은 속+같은 종 epithet 일치"만 정탐으로 인정하도록 재설계했습니다.

## 4. 데이터 검증 자동화

그룹/종 데이터 전체에 대해 아래 규칙을 매번 자동 검사합니다.

- 범위 역전 (예: 최소값 > 최대값)
- 허용되지 않는 enum 값 (temperament, difficulty 등)
- 끊어진 외래키 참조 (`species.groupId`가 존재하지 않는 그룹을 가리키는 경우)
- 중복 ID

## 5. ETL 파이프라인 (Excel ↔ JSON)

비개발자도 데이터를 검토·수정할 수 있도록 Excel 왕복 파이프라인을 만들었습니다.

- `scripts/export_to_excel.py`: JSON → 사람이 읽기 좋은 Excel (종 시트 + 케어그룹 시트 + 합사예외 시트, 드롭다운 검증 포함)
- `scripts/reimport_from_excel.py`: 수정된 Excel → JSON, 위 4번 검증 규칙을 통과해야 반영됨
- 라운드트립(JSON → Excel → JSON) 테스트로 변환 과정에서 데이터 손실이 없는지 확인

## 6. 의사결정 엔진 설계 원칙

- 숫자 데이터(온도·pH)는 **구간 겹침 계산**
- 정성 데이터(성격)는 **카테고리 매트릭스 룩업** — 임의의 점수화 금지
- 보조 요인은 **점수 합산이 아니라 단계적 다운그레이드** — "왜 이 등급이 나왔는지" 항상 역추적 가능하게 유지
- 규칙으로 못 잡는 예외는 **별도 예외 테이블**로 관리하고, 출처를 명시
